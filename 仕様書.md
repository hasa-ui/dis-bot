# 仕様書

## 1. 概要

この Bot は、Discord サーバーごとに「ステータス段階」を設定し、対象メンバーへ対応するロールを付与・遷移・解除するための管理 Bot である。

現在の公開仕様は、固定 3 段の「違反ロール」モデルではなく、guild ごとに可変な段階数を持つ「ステータス段階」モデルを前提とする。旧 `light` / `medium` / `heavy` ベースのデータは、起動時に新モデルへ自動移行される legacy 互換としてのみ扱う。

主な機能は以下。

- guild ごとの段階数設定
- 各段階のロール、期間、表示名、満了時動作の設定
- 対象メンバーへのステータス付与、確認、即時解除
- 満了時の自動遷移
- メンバー再参加時のロール再付与
- SQLite への状態保存
- Termux 上での自己更新付き起動および supervisor 運用

## 2. 用語

- `段階`: ステータスの強さを表す単位。`stage_index` が大きいほど強い段階とみなす。
- `段階設定`: 各段階に紐づくロール、期間、表示名、満了時動作。
- `active record`: 現在メンバーに適用されている `status_records` の 1 行。
- `満了時動作`: 段階の期限が来たときの遷移方法。`next` / `clear` / `hold` のいずれか。
- `legacy migration`: 旧 `guild_settings` / `sanctions` から新モデルへ変換する処理。

## 3. 公開コマンド仕様

すべての slash command は guild 内でのみ利用できる。DM では利用できない。

### 3.1 `/setup`

サーバーのステータス設定をまとめて行うための入口コマンド。

- 権限:
  - 実行時に `Manage Server` または administrator が必要
- 応答:
  - ephemeral な setup UI を表示する
  - 現在の設定状態、段階数、各段階の要約、未設定項目を表示する
- 主な用途:
  - 段階数設定
  - 段階編集
  - 現在設定の再表示

### 3.2 `/status_config`

現在のステータス設定を表示する。

- 権限:
  - 特別な管理権限は不要
- 応答:
  - 現在の設定状態
  - 段階数
  - 各段階のロール / 期間 / 満了時動作
  - 未設定項目

### 3.3 `/status_set member stage reason?`

対象メンバーへ指定段階のステータスを付与または上書きする。

- 権限:
  - `Manage Roles`
- 引数:
  - `member`: 対象メンバー
  - `stage`: 1〜10 の整数
  - `reason`: 任意文字列
- 実行条件:
  - guild に設定が存在すること
  - 指定 `stage` が guild の `stage_count` 以内であること
  - 指定段階から実際に到達する経路上の段階設定が完了していること
  - Bot が対象を操作可能であること
    - サーバーオーナー不可
    - Bot 自身不可
    - Bot より上位または同位ロールの相手不可
- 成功時:
  - `status_records` を作成または更新する
  - 対応ロールを付与し、他のステータスロールは外す
  - 次回変更予定と理由を返す
- 失敗例:
  - `Manage Roles` 不足
  - Bot のロール順不足
  - 設定未完了
  - 段階番号が guild の段階数を超える

### 3.4 `/status_clear member`

対象メンバーのステータスロールを即時解除する。

- 権限:
  - `Manage Roles`
- 動作:
  - 該当メンバーの `status_records` を削除する
  - ステータスロールをすべて外す

### 3.5 `/status_view member`

対象メンバーの現在状態を確認する。

- 権限:
  - 特別な管理権限は不要
- 動作:
  - まず対象 record を reconcile する
  - active record が残っていれば、現在段階、次回変更、理由を返す
  - active record がなければ「有効なステータス状態はありません」と返す

## 4. 設定モデル

### 4.1 guild ごとの設定

各 guild は 1〜10 段階の設定を持つ。

- `stage_count`: 現在有効な段階数
- 各段階の設定:
  - `stage_index`
  - `label`
  - `role_id`
  - `duration_seconds`
  - `on_expire_action`

### 4.2 段階の意味

- `stage_index` は 1 以上の整数
- 数字が大きい段階ほど強い
- 既定表示名は `段階{n}`
- `label` が空でなければ表示は `段階{n}（label）`

### 4.3 満了時動作

- `next`
  - 1 つ弱い段階へ移行する
  - 移行先段階の duration を現在の `expires_at` に加算する
- `clear`
  - record を削除し、ロールを解除する
- `hold`
  - 同じ段階を維持する
  - `expires_at = NULL` に変更し、以降は自動満了しない

### 4.4 入力制約

- 段階数は 1〜10
- 期間は 1〜3650 日
- 各段階のロールは必須
- 同一 guild 内で同じロール ID を複数段階に使えない
- `stage 1` で `next` は選べない

## 5. `/setup` UI 仕様

`/setup` は ephemeral な対話 UI で設定を行う。

### 5.1 画面の性質

- コマンド実行者本人のみ操作できる
- follow-up 操作でも毎回 `Manage Server` または administrator を再確認する
- timeout は 600 秒

### 5.2 Home 画面

表示内容:

- 設定状態（完了 / 未完了）
- 段階数
- 各段階の要約
- 未設定項目

ボタン:

- `段階数設定`
- `段階編集`
- `再表示`

### 5.3 段階数設定

モーダルで段階数を入力する。

- 入力範囲は 1〜10
- 保存時は interaction を defer してから処理する
- 保存後は既存 active record に対してロール再適用を行う
- 段階数を減らす場合、削除対象段階の record は新しい最大段階へ丸める

### 5.4 段階編集

1 段階ずつ以下を編集する。

- ロール
- 表示名
- 期間（日数）
- 満了時動作

保存時の動作:

- 現在の guild 設定との整合性を検証する
- 変更前に使われていたステータスロールも除去対象に含める
- 既存 active record に対してロール再適用を行う

## 6. 状態遷移仕様

### 6.1 record 作成

`/status_set` 成功時は、対象メンバーに対して 1 guild 1 record の `status_records` を upsert する。

- `stage_index`: 指定段階
- `expires_at`: `now + current_stage.duration_seconds`
- `reason`: 指定理由

### 6.2 満了処理

Bot は 1 分ごとのループで、期限到来した record を処理する。

- `expires_at IS NULL` の record は満了処理しない
- `clear`:
  - record 削除
  - ロール解除
- `hold`:
  - `expires_at = NULL`
  - 同じ段階のロールを維持
- `next`:
  - `stage_index - 1` へ移行
  - 新しい `expires_at = 旧 expires_at + 次段階 duration`
  - 対応ロールを更新

### 6.3 再参加時

メンバーが再参加したとき、active record があれば reconcile 後に現在段階のロールを再付与する。

### 6.4 段階数縮小時

guild の `stage_count` を減らした場合、上位段階の active record は新しい最大段階へ丸める。

- `stage_index` を新最大段階へ変更する
- `expires_at` は新最大段階の duration を基準に再計算する
- 既存の古いステータスロールは除去対象に含める

## 7. ロール適用仕様

ロール付与時は、現在設定済みの全ステータスロールを一度除外し、必要なら新しい段階ロールだけを追加する。

- guild 設定変更時は、変更前に存在したロール ID も除去対象へ含める
- `guild.get_role(role_id)` でロールが見つからない場合は実行エラー
- `Manage Roles` 不足やロール順不足で `member.edit()` が失敗した場合はコマンド側でエラー応答する

## 8. データ保存仕様

既定 DB パスは `violations.db`。実体は SQLite。

### 8.1 現行テーブル

- `guild_status_settings`
  - `guild_id`
  - `stage_count`
  - `updated_at`
- `guild_status_stages`
  - `guild_id`
  - `stage_index`
  - `label`
  - `role_id`
  - `duration_seconds`
  - `on_expire_action`
  - `updated_at`
- `status_records`
  - `guild_id`
  - `user_id`
  - `stage_index`
  - `expires_at`
  - `reason`
  - `updated_at`

### 8.2 legacy テーブル

以下の旧テーブルは migration 用に残る。

- `guild_settings`
- `sanctions`

### 8.3 起動時 migration

新テーブル側に guild 設定が存在しない場合のみ、旧データを取り込む。

- `light -> stage 1`
- `medium -> stage 2`
- `heavy -> stage 3`

旧段階の満了時動作は次の対応で移行する。

- stage 1: `clear`
- stage 2: `next`
- stage 3: `next`

`sanctions` の active data は `status_records` へコピーされる。

## 9. 起動・運用仕様

### 9.1 `bot.py`

`bot.py` は薄いエントリポイントであり、`status_bot.create_bot()` で Bot を作成し、`DISCORD_TOKEN` を使って起動する。

### 9.2 `runbot.sh`

直接起動用スクリプト。

- `origin/main` を fetch する
- `main` への checkout を試みる
- 条件を満たせば `git reset --hard origin/main` で自己更新する
- `setenv.sh` を読み込んで `bot.py` を起動する

非 `main` ブランチで checkout が失敗した場合は、別ブランチ保護のため起動を中止する。

### 9.3 `supervisor.sh`

監視・再起動・更新検知用スクリプト。

- 起動時に remote 更新を確認する
- deploy 成功時のみ bot を stop/start する
- 1 分間隔で更新検知と crash recovery を行う
- `LAST_SEEN` を last-known-good revision として保持する
- current-mode 再起動は `LAST_SEEN` と一致し、かつ `main` ブランチ上の checkout に限定する
- current-mode の reset 失敗は supervisor を落とさず、その回の再起動だけをスキップする

### 9.4 環境変数

- `DISCORD_TOKEN`: 必須
- `DB_PATH`: 任意。未指定時は `violations.db`

運用用の実値は `setenv.sh` に置く。repo 内には `setenv.example.sh` がテンプレートとして存在する。

## 10. 権限と前提

- `/setup` は `Manage Server` または administrator が必要
- `/status_set` と `/status_clear` は `Manage Roles` が必要
- Bot 自身にも `Manage Roles` と十分なロール順が必要
- Bot はサーバーオーナー、Bot 自身、Bot より上位または同位ロールの相手を変更できない

## 11. 制約・補足

- slash command の `stage` は guild ごとの動的 choices ではなく、1〜10 の整数入力で受ける
- 実機の Discord 権限状態やロール階層によっては、設定が正しくてもロール変更に失敗しうる
- `hold` 段階は自動解除されない
- 仕様上の表示文言は日本語ベースである
- 本仕様書は現行実装ベースであり、将来案や未実装機能は含まない
