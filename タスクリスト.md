# タスクリスト

## 概要

この文書は `ロードマップ.md` を実行用タスクへ落とし込んだものである。今回の実行対象は中期ロードマップとし、短期ロードマップは完了済みの履歴として残す。優先順は「日常作業の削減」「設定再利用性の向上」「初期セットアップの簡略化」「理由記録の品質安定化」に合わせて整理する。

## 既存公開 I/F

- 維持する既存コマンド: `/setup` / `/status_config` / `/status_notify_config` / `/status_set` / `/status_clear` / `/status_view` / `/status_list` / `/status_history`
- 中期追加候補: `/status_bulk_set` / `/status_bulk_clear` / `/status_export` / `/status_import` / `/status_template_apply` / `/status_reason_config`
- 既存 UI を前提に扱うもの: `/setup` 保存前プレビュー、通知設定、履歴表示、一覧表示

## 完了済み短期タスク

- [x] `/setup` 保存前プレビューを追加済み。保存直前に再適用対象人数・丸め対象人数・見つからないロール件数を再計算して表示できる。
- [x] `/status_list` を追加済み。guild 単位の active record を一覧表示し、表示量が多い場合はページングする。
- [x] 履歴・監査ログを追加済み。active record と分離した履歴保存先を持ち、`/status_history` で主要イベントを参照できる。
- [x] 通知設定を追加する。目的: 自動遷移や解除の見落とし防止。主な変更箇所: `status_bot/store.py` / `status_bot/service.py` / `status_bot/commands.py`。完了条件: guild ごとに通知先チャンネルと通知対象イベントを保持し、手動付与・手動解除・自動遷移・`hold` 移行・設定変更の通知可否を制御できる。

## 中期実行タスク

### 5.1 一括操作

- [x] `/status_bulk_set` と `/status_bulk_clear` の v1 仕様を固定する。入力方式は添付テキストのみに限定し、1 行 1 メンバー ID または mention を受け付ける。複数メンバー直接指定とロール保有者単位の一括操作は v1 対象外と明記する。
- [x] 一括入力の前処理を実装する。重複行、空行、未知メンバー、guild 外メンバー、Bot 自身、管理対象外メンバーを個別に分類し、実行前サマリーを返せるようにする。
- [x] `/status_bulk_set` を追加する。既存 `assign_status()` を再利用し、指定段階と理由を全対象へ順次適用する。部分失敗は全体失敗にせず、成功件数・失敗件数・失敗理由内訳を返す。
- [x] `/status_bulk_clear` を追加する。既存 `clear_status()` を再利用し、active record なし・権限不足・ロール変更失敗をメンバー単位で集計する。
- [x] 一括操作の実行制御を固める。1 件ずつ await し、Discord rate limit を避ける。通知は既存単体操作と同じ条件で発火させ、バルク専用通知は追加しない。
- [x] 一括操作の検証項目を追加する。正常系、重複入力、未知メンバー混在、部分失敗、全失敗、通知 ON/OFF 時の挙動を確認する。
- [x] 完了条件: 添付テキストだけで複数メンバーの付与・解除ができ、部分失敗を含む結果要約と回帰テストを揃えた状態にする。

### 5.2 設定 export / import

- [x] export / import の JSON 形式を固定した。v1 の対象は guild の段階設定のみとし、`stage_count` と各段階の `role_id` `duration_days` `expiry_action` を含める。active record、history、notification config、理由プリセットは含めない。
- [x] `/status_export` を追加した。現行 guild 設定を JSON 化して file attachment で返し、変更前バックアップ用途でそのまま保存できる形にした。
- [x] `/status_import` の事前 validation を実装した。JSON 構文、段階数範囲、各段階の必須項目、`expiry_action` の妥当性、role ID の存在確認を行う。
- [x] `/status_import` を追加した。validation 成功時だけ設定へ反映し、上書きされる設定の要約を返す。active record と履歴は保持したまま、必要な再適用だけ行う。
- [x] import 前プレビューを追加した。現設定との差分、見つからない role 数、反映予定段階数を保存前に表示する。
- [x] export / import の検証項目を追加した。正常系、不正 JSON、不足 role、欠損 stage、既存設定上書き、legacy migration 非影響を確認する。
- [x] 完了条件: guild の段階設定だけを JSON で安全に書き出し・取り込みでき、上書き前 validation とプレビューを備えた状態にした。

### 5.3 テンプレート適用

- [x] v1 テンプレートを固定する。`3段標準` と `4段警告強化型` の 2 種を用意し、段階数・期間・満了時動作を文書化する。role はテンプレートに含めず、適用後に guild ごとに設定する前提にする。
- [x] `/status_template_apply` の preview 内容を定義する。適用予定の段階数、各段階の期間、各段階の `expiry_action`、既存設定との差分を表示する。
- [x] `/status_template_apply` を追加する。適用時は既存 `save_stage_count_settings()` と `save_stage_settings()` を再利用し、通知は既存の設定変更通知に統一する。
- [x] テンプレート適用時の active record 影響確認を実装する。段階数縮小や満了時動作変更で影響を受ける件数を保存前プレビューへ含める。
- [x] テンプレート適用の検証項目を追加する。preview のみ、適用実行、既存 active record あり、通知 ON/OFF 時の挙動を確認する。
- [x] 完了条件: 固定テンプレート 2 種を preview 付きで適用でき、既存設定との差分と active record 影響を事前確認できる状態にする。

### 5.4 理由入力の標準化

- [ ] guild 単位の理由プリセット保存仕様を固定する。v1 は補助機能とし、自由入力は残す。プリセット件数上限と 1 件あたりの文字数上限を明記する。
- [ ] 理由プリセット管理コマンド `/status_reason_config` を追加する。一覧表示、追加、更新、削除を guild 単位で扱えるようにする。
- [ ] `/status_set` から理由プリセットを利用できるようにする。プリセット選択のみ、自由入力のみ、プリセット + 補足自由入力の 3 パターンを許可する。
- [ ] 理由保存と通知表示の扱いを揃える。履歴保存時は最終的な理由文字列を保存し、通知は既存の 2000 文字制限に収まる形を維持する。
- [ ] 理由入力標準化の検証項目を追加する。preset 選択、自由入力併用、長文理由、preset 削除後、通知文面の文字数制限を確認する。
- [ ] 完了条件: guild ごとの理由プリセットを管理でき、`/status_set` から自由入力を残したまま再利用可能な理由入力補助を提供できる状態にする。
