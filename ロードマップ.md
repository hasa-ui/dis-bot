# ロードマップ

## 1. 目的

この文書は、現行のステータス段階 Bot をさらに便利にするための改善候補を整理するためのロードマップである。

現時点の主眼は次の 2 つ。

- 日常運用を楽にすること
- ステータスモデルの表現力を広げること

この文書は将来計画であり、ここに書かれた項目は未実装を含む。実装済みの仕様そのものはコードを基準とし、ここでは「現在地」と「次の候補」を区別して扱う。

## 2. 現在地

現在、Bot には以下が実装されている。

- guild ごとの可変段階ステータス設定
- 各段階のロール / 期間 / 表示名 / 満了時動作 (`next` / `clear` / `hold`)
- `/setup` による設定 UI
- `/status_config` / `/status_set` / `/status_clear` / `/status_view`
- 満了時の自動遷移
- 再参加時のロール再付与
- SQLite 保存と legacy migration
- `runbot.sh` / `supervisor.sh` による Termux 運用

一方、運用上は次の不足が残っている。

- active record の一覧表示がない
- 変更履歴や監査ログがない
- 自動遷移や解除の通知がない
- 一括操作がない
- 設定変更前に影響件数を見積もりにくい
- guild 間で設定を再利用しにくい
- 現在段階を維持したまま期限だけ調整する手段がない

## 3. 優先順位の考え方

優先順位は次の順で決める。

1. 運用事故を減らすもの
2. 日常の手作業を減らすもの
3. モデルの表現力を広げるもの
4. 実装コストが重い周辺機能

前提として、次は崩さない。

- guild 単位設定
- ロールは ID 基準
- Termux 運用と `runbot.sh` / `supervisor.sh` の前提
- SQLite ベースの永続化
- 既存コマンド互換の維持

## 4. 短期ロードマップ

### 4.1 ステータス一覧

- 目的:
  - 現在 active なステータス保持者をすぐ把握できるようにする
- 追加案:
  - `/status_list`
- 利用者価値:
  - 手動で個別確認しなくても、現在の適用状況を確認できる
- 実装メモ:
  - `status_records` を guild 単位で取得し、段階別または期限順で表示する
  - 表示量が多い場合はページングや件数制限を入れる
- 工数感:
  - 小

### 4.2 履歴・監査ログ

- 目的:
  - 誰がいつ何を付与・解除・変更したかを追えるようにする
- 追加案:
  - `/status_history member`
  - 設定変更や自動遷移も履歴化
- 利用者価値:
  - 後追い確認、引き継ぎ、トラブル解析がしやすくなる
- 実装メモ:
  - 新しい履歴テーブルを追加し、手動操作・自動遷移・設定変更を記録する
  - 既存 active record とは別管理にする
- 工数感:
  - 中

### 4.3 通知設定

- 目的:
  - 自動遷移や解除を見落としにくくする
- 追加案:
  - `/status_notify_config`
- 利用者価値:
  - 管理チャンネルで変化を追える
- 実装メモ:
  - guild ごとに通知先チャンネルと通知対象イベントを設定する
  - 通知対象候補:
    - 手動付与
    - 手動解除
    - 自動遷移
    - `hold` への移行
    - 設定変更
- 工数感:
  - 中

### 4.4 設定変更プレビュー

- 目的:
  - `/setup` 保存前に影響を把握できるようにする
- 追加案:
  - 保存前のプレビュー表示
- 利用者価値:
  - 段階数変更やロール差し替え時の事故を減らせる
- 実装メモ:
  - 再適用対象人数、丸め対象人数、見つからないロール件数などを表示する
  - 保存直前に再計算して出す
- 工数感:
  - 小

## 5. 中期ロードマップ

### 5.1 一括操作

- 目的:
  - 複数メンバーへの付与・解除をまとめて行えるようにする
- 追加案:
  - `/status_bulk_set`
  - `/status_bulk_clear`
- 利用者価値:
  - 手動作業を大幅に減らせる
- 実装メモ:
  - メンバー複数指定、ロール保有者単位、あるいは添付テキスト入力などの操作方法を別途決める
  - rate limit と部分失敗の扱いを明示する
- 工数感:
  - 中

### 5.2 設定 export / import

- 目的:
  - guild 設定のバックアップと再利用を可能にする
- 追加案:
  - `/status_export`
  - `/status_import`
- 利用者価値:
  - 複数サーバーで同じ設定を再現しやすい
  - 変更前バックアップにも使える
- 実装メモ:
  - JSON ベースのやり取りを想定
  - role ID の存在確認と import 時バリデーションが必要
- 工数感:
  - 中

### 5.3 テンプレート適用

- 目的:
  - 初期設定を簡単にする
- 追加案:
  - `/status_template_apply`
- 利用者価値:
  - 小規模サーバーでもセットアップしやすい
- 実装メモ:
  - 3 段標準、4 段警告強化型など、いくつかのテンプレートを持つ
  - 実適用前にプレビューする
- 工数感:
  - 小〜中

### 5.4 理由入力の標準化

- 目的:
  - 理由の記録品質をそろえる
- 追加案:
  - 理由プリセット
  - 定型文補助
- 利用者価値:
  - 履歴が読みやすくなる
  - 運用者間の表記揺れを減らせる
- 実装メモ:
  - guild ごとの理由テンプレートを追加
  - 自由入力は残す
- 工数感:
  - 中

## 6. 長期ロードマップ

### 6.1 任意遷移

- 目的:
  - `next` 固定より柔軟な遷移を可能にする
- 追加案:
  - 満了時に任意の段階へ移行
- 利用者価値:
  - 2 段飛ばしや特別フローを表現できる
- 実装メモ:
  - `on_expire_action` を単純 enum から遷移先指定モデルへ広げる必要がある
  - 無限ループ防止バリデーションが必要
- 工数感:
  - 大

### 6.2 例外ルール

- 目的:
  - 特定メンバーや条件に対して標準フローを外せるようにする
- 追加案:
  - 対象外ロール
  - 再付与除外
  - 通知除外
- 利用者価値:
  - 実運用の細かな例外に対応しやすい
- 実装メモ:
  - guild 全体ルールと member 個別例外のどちらを採るかを分けて設計する必要がある
- 工数感:
  - 大

### 6.3 期限だけの調整

- 目的:
  - 現在段階を変えずに期限延長 / 短縮を行えるようにする
- 追加案:
  - `/status_extend`
  - `/status_shorten`
- 利用者価値:
  - 付与し直さずに柔軟な調整ができる
- 実装メモ:
  - 履歴と通知の整合を前提に設計する
- 工数感:
  - 中

### 6.4 閲覧 UI 強化

- 目的:
  - 設定や current state を視覚的に理解しやすくする
- 追加案:
  - 遷移フロー要約
  - 影響対象数サマリー
  - 一覧系のページ UI
- 利用者価値:
  - 管理者が設定の意味を把握しやすくなる
- 実装メモ:
  - Discord message の文字数と操作性の制約内で組む
- 工数感:
  - 中〜大

## 7. 検討のみ

以下は便利になりうるが、現時点では優先度や実装コストの観点から採用未定とする。

### 7.1 DM 通知

- 利点:
  - 対象者へ直接通知できる
- 懸念:
  - DM 拒否や言い回し管理が必要

### 7.2 Web UI

- 利点:
  - 一覧・履歴・設定編集を見やすくできる
- 懸念:
  - Termux 運用の複雑化
  - 認証と公開範囲の設計が必要

### 7.3 動的 slash command choices

- 利点:
  - guild ごとの段階名を入力補助に使える
- 懸念:
  - 実装と同期コストが増える
  - 現状の整数入力でも機能上は足りている

### 7.4 複数設定セットの同居

- 利点:
  - 用途別に複数のステータス系統を持てる
- 懸念:
  - データモデルと UI が大きく複雑化する

## 8. 実施順の提案

最初の実施順は次で固定する。

1. `/status_list`
2. 履歴・監査ログ
3. 通知設定
4. `/setup` の設定変更プレビュー
5. 一括操作
6. export / import
7. テンプレート適用
8. 期限だけの調整
9. 任意遷移
10. 例外ルール

## 9. 補足

- このロードマップは「便利にするための候補整理」であり、実装確約ではない
- 各項目の詳細仕様は、着手時に別途仕様化する
- 既存コマンド名や保存モデルを壊す変更は、互換方針を決めてから進める
